# === FINDINGS ===
#
# Q6 ✅ ANSWERED in two parts.
#
# ─── view.applyPropertyUpdates(payload) — VIEW-LEVEL (we got the source) ───
#
# Deminified source:
#
#   applyPropertyUpdates(e) {
#     Object.keys(e).forEach((t) => {
#       try {
#         if ("" === t) {
#           const inner = e[""];
#           inner.params && this.params.update(inner.params);
#           inner.props  && this.props.update(inner.props);
#           inner.custom && this.custom.update(inner.custom);
#         } else {
#           const path = t.split(":").map((s) => parseInt(s));
#           this.root.dispatchUpdates(path, e[t]);
#         }
#       } catch (err) {
#         h.fatalException(`Error applying property updates on ${this.instanceId}.${t}`, err);
#       }
#     });
#   }
#
# Payload shape: an OBJECT (not array!) whose keys are either:
#   • ""              — bulk update for view's params / props / custom property trees.
#                       Value: { params: <encoded-tree>, props: <encoded-tree>, custom: <encoded-tree> }
#                       Each non-empty subkey is forwarded to that tree's .update() method.
#   • "1:2:3"         — colon-separated NUMERIC path. Forwarded to
#                       this.root.dispatchUpdates(parsedPath, value) for component-tree updates.
#                       The numbers are component IDs in the root composite tree.
#
# So a typical synthesized call would be:
#
#   ourView.applyPropertyUpdates({
#     "": {
#       "params": {"$":["map/merge"], "color": {"$":["str","#FF0000"]}}
#     }
#   });
#
# This DIRECTLY closes Q2 (params.update encoded format): it's the same
# tuple-tagged tree we saw in the client-value-update WS frame in Probe 3.
#
# Why the array calls "OK"d but didn't apply: Object.keys([{...}]) returns ["0"].
# "0".split(":").map(parseInt) = [0]. dispatchUpdates([0], <obj>) was attempted
# but wrapped in try/catch, so silently swallowed. Not an error; just no-op
# because the value shape didn't match what dispatchUpdates expects.
#
# ─── page.applyPropertyUpdates(frame) — SESSION-LEVEL (native, inferred) ───
#
# Native code, no source. But the error messages reveal the expected shape:
#   • null              → "Cannot read properties of null (reading 'values')"
#                         → expects an object with a `.values` field
#   • [] / [{...}]      → "Cannot read properties of undefined (reading 'props')"
#                         → walks `.values.<something>.props` and fails
#
# Cross-reference Probe 3's captured incoming WS frame:
#   { "values": { "views": {}, "page": {}, "session": { "props": {...} } },
#     "sync": [] }
#
# So page.applyPropertyUpdates EXPECTS the full incoming-WS-frame envelope.
# It's literally the entry point for replaying gateway-pushed updates.
#
# Synthesizing one would look like:
#
#   window.__client.page.applyPropertyUpdates({
#     "values": {
#       "views":   { "<mountPath>": <view-level-payload> },
#       "page":    {},
#       "session": {}
#     },
#     "sync": []
#   });
#
# (Exact shape of the per-view inner payload still needs confirmation — but
# given the symmetry with view.applyPropertyUpdates, it's likely keyed the
# same way: "" for bulk + "1:2:3" for component paths.)
#
#
# ─── page.applyPendingUpdates() — bonus ───
#
# No-arg method. Returns OK. Drains the page.pendingPropertyUpdates queue
# (which we saw exists as a session field in Probe 1). Useful if we ever
# want to queue updates and apply them all at once.
#
